Подтверждено улучшение Без резервного URL все 1028 affected-цепочек остались бы на ошибке primary-хоста. После релиза 695 из них получили успешный SOAP-ответ на backup-хосте, то есть ошибка была предотвращена в двух третях случаев.
Остаточный риск Failover не делает интеграцию независимой от MAKC: если backup-хост тоже возвращает 500/502/503, цепочка все равно остается ошибочной. Это наблюдалось на 333 цепочках.

Методика и SQL-смысл проверки

Проверка выполнялась read-only по production log DB. Основной принцип: сравнить запросы MAKC SOAP по группам ApplicationId + TraceId, где primary host дал ошибку, а затем был вызван backup host.

-- SQL intent, сокращенно:
-- 1. Отобрать MaksLogs с Date >= '2026-07-07T19:18:00+03:00'
-- 2. Оставить SOAP XmlDocumExchangeWS URL.
-- 3. Сгруппировать по ApplicationId, TraceId.
-- 4. Primary failure = primary URL + ResponseCode not like '200%' / null / body without <return>.
-- 5. Recovered = после primary failure есть backup URL + 200 OK + body with <return>.
-- 6. Проверить BusinessLogs и ErrorLogs по distinct ApplicationId из failover cohort.

Cross-DB join из Core к InsappLogProd.dbo.MaksLogs был недоступен из-за Invalid object name, поэтому детальный статусный срез по Core для failover cohort в этот отчет не включен. Логовый анализ выполнен по MaksLogs, BusinessLogs, ErrorLogs.

Финальный вывод Доработка с резервным хостом улучшила ситуацию: при ошибках primary MAKC примерно две трети affected-цепочек были восстановлены успешным ответом backup-хоста. Механика переключения работает. Оставшийся дефектный хвост связан не с отсутствием переключения, а с нестабильностью самого backup-хоста в отдельных временных окнах.
